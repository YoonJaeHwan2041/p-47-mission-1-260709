# 고민 사항

### 1. 데이터를 어떻게 저장할까 — String, 2차원 배열, 객체

처음엔 `ArrayList<String>`에 "명언 - 작가"처럼 문자열을 합쳐서 저장하려 했다.
하지만 나중에 명언과 작가를 각각 따로 꺼내 쓰려면 문자열을 다시 쪼개야 해서
번거롭고 실수하기 쉽다는 문제가 있었다.

2차원 배열(`String[][]`)도 고려했지만:
- 크기를 미리 정해야 해서 "몇 개가 등록될지 모른다"는 요구사항과 안 맞음
- `arr[0][0]`, `arr[0][1]`처럼 인덱스로 접근하면 그게 명언인지 작가인지
  코드만 봐서는 알기 어려움
- 필드마다 타입이 다르면(예: 등록 날짜 추가) 애초에 사용 불가

**결론:** 명언 + 작가처럼 의미가 다른 값이 한 세트로 묶이고, 개수도 계속
늘어나는 상황이므로 **클래스(객체) + `ArrayList`** 조합이 가장 적합하다고 판단.

```java
class WiseSaying {
    int id;
    String saying;
    String author;
}
ArrayList<WiseSaying> wiseSayings = new ArrayList<>();
```

### 2. id는 어떻게 매길까 — 인덱스 vs 고유 번호

처음엔 목록 출력 시 `for`문의 인덱스(`i+1`)를 번호로 썼다. 등록 순서와
리스트 인덱스가 항상 일치하기 때문에 문제없이 동작했다.

하지만 **삭제 기능이 생기면 얘기가 달라진다.** 중간 항목을 삭제하면 뒤 항목들의
인덱스가 자동으로 당겨지므로, "고유 번호"가 아니라 "그때그때 바뀌는 위치"가
되어버린다. 나중에 파일이나 DB에 저장할 것까지 고려하면 삭제해도 변하지 않는
고유 id가 필요하다고 판단했다.

**해결:** 클래스에 `static` 카운터를 두어 DB의 AUTO_INCREMENT와 비슷한 효과를 구현.

```java
class WiseSaying {
    static int nextId = 1; // 모든 객체가 공유하는 값 (static)
    int id;                // 객체마다 독립적으로 갖는 값

    WiseSaying(String saying, String author) {
        this.id = nextId;
        nextId++;
        ...
    }
}
```

- 일반 필드는 객체마다 독립된 저장공간을 가짐
- `static` 필드는 클래스 전체를 통틀어 하나만 존재하고, 모든 객체가 공유해서 봄
- 프로그램이 켜져 있는 동안 유지되며, 특정 객체가 사라져도 static 값은 영향받지 않음

### 3. 자료구조 재검토 — ArrayList vs HashMap

id로 검색/삭제할 일이 많아지면서, "id를 key로 하는 `HashMap`을 쓰면 매번
리스트를 순회하지 않아도 되니 더 빠르지 않을까"라는 고민을 했다.

| 자료구조 | 순서 유지 | id로 검색 속도 |
|---|---|---|
| `ArrayList` | O (등록 순서) | 느림 (순회 필요) |
| `HashMap` | X (순서 보장 안 됨) | 빠름 |
| `LinkedHashMap` | O (등록 순서) | 빠름 |

`HashMap`은 해시값 기준으로 저장되기 때문에 순회 시 등록 순서가 보장되지
않는다는 트레이드오프가 있다 (순서까지 유지하려면 `LinkedHashMap`이 필요).

**결론:** 지금은 데이터 양이 많지 않은 콘솔 과제 수준이라 성능 차이가
체감되지 않고, 목록을 등록 순서대로 보여줘야 하는 요구사항이 있어
**`ArrayList`를 그대로 유지**하기로 함. 자료구조 선택 시 "저장 순서 유지가
필요한가"와 "검색 빈도/양"을 함께 고려해야 한다는 걸 확인한 과정.

### 4. 명령어 파싱 — `삭제?id=1` 형태 처리

명령어와 파라미터가 섞인 입력(`삭제?id=1`)을 처리하기 위해 문자열을
쪼개는(parsing) 과정이 필요했다.


### 5. switch문 fall-through 트러블슈팅

`case` 로직이 끝난 뒤 `break`를 빠뜨려서, 의도치 않게 다음 `case`까지 실행되는
문제를 두 번 겪었다.

- **1차 증상:** `수정` 처리 후 `default`("잘못된 명령어입니다")까지 같이 출력됨
  → `case "수정":` 블록 끝에 `break`가 없어서 발생
- **2차 증상:** `if(found == false) { ...; break; }`처럼 `break`를 조건문 안에만
  넣어서, 성공 케이스(`found == true`)일 때는 여전히 다음 case로 흘러감
  → `break`는 "가장 가까운 반복문/switch 하나만" 탈출한다는 점을 이해하고,
  `for`문용 `break`(반복 중단)와 `switch`용 `break`(case 종료)를 구분해서
  각각 필요한 위치에 배치하도록 수정

### 6. 역할 분리 — Main(컨트롤러) / Service(로직)

기능이 늘어나면서 `Main`의 `switch`문 안에 입력 처리, 데이터 검색, 조건 분기가
전부 뒤섞여 가독성이 떨어지는 문제가 생겼다. 무조건 메서드로 잘게 쪼개는 것도
정답은 아니라는 걸 느꼈다 — switch는 깔끔해져도 결국 로직이 다른 곳으로
이동할 뿐, "중복이 몇 번 이상일 때만 분리한다"는 기준(Rule of Three)으로
판단하는 게 낫다고 결론지었다.

최종적으로 아래 기준으로 책임을 나누기로 함:

| 계층 | 역할 |
|---|---|
| **Main** (컨트롤러) | 사용자 입력받기(Scanner), 명령어 분기(switch), Service 호출, 결과 메시지 출력 |
| **Service** | 데이터 저장/조회/삭제/수정 로직, 성공 여부를 리턴값으로만 알려줌 (입출력 없음) |

```java
// Service - 순수 로직만, 입출력 없음
boolean delete(int id) {
    for (...) {
        if (...) { wiseSayings.remove(i); return true; }
    }
    return false;
}

// Main - 입력받고, Service 호출하고, 결과만 출력
boolean success = service.delete(id);
System.out.println(success ? id + "번째 삭제되었습니다." : id + "번을 찾지 못했습니다.");
```

이렇게 나누면 Service는 입출력 방식(콘솔/웹 등)이 바뀌어도 그대로 재사용할 수
있다는 장점이 있다. 이 구조는 이후 스프링부트에서 배우게 될
Controller-Service 계층 구조와 개념적으로 동일하다는 점을 미리 확인했다.

## 배운 것 정리

- 배열은 `.length`(필드), `String`은 `.length()`(메서드), `ArrayList`는 `.size()`(메서드) — 헷갈리기 쉬운 포인트
- 지역 변수는 초기화 전 사용 시 컴파일 에러, 클래스 필드는 자동으로 기본값(`0`, `false`, `null`) 할당
- `static`은 "객체 단위가 아니라 클래스 단위로 하나만 존재"하는 값 — 여러 객체가 공유
- `switch`의 `break`는 가장 가까운 반복문/switch 하나만 탈리출 (fall-through 주의)
- 자료구조 선택은 "성능"만이 아니라 "순서 유지가 필요한가"까지 함께 고려해야 함
